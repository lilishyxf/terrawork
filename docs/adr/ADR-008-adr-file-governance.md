# ADR-008: ADR 文件治理一致性策略

## 背景

M0 阶段定型时，ADR-001~005 内联写在 ARCHITECTURE.md 第 3 节「核心设计决策记录」内；M0 末期与 M1.1 阶段新增的 ADR-006 / ADR-007 抽成 `docs/adr/` 下的独立 markdown 文件。这种"部分内联、部分独立"的状态在 git init 准备首次提交时被识别为隐性裂缝——每次新增 ADR 都要做一次微决策（独立成文件 vs 塞回 §3），无统一规则会持续摇摆，且容易在后期 review 中被错误地"修整一致"导致大规模返工。

## 决策

采用混合治理 + 时间分界规则：

1. **M0 基线性 ADR（ADR-001~005）保持内联**在 ARCHITECTURE.md §3，不抽取。它们是架构主结构的骨架决策，与 ARCHITECTURE.md 主文同时定型，内联保证 M0 基线"读完一个文件就理解架构全貌"的自包含特性。

2. **M0 末期起所有新增 ADR（ADR-006 起）独立成文件**，位于 `docs/adr/`，命名规则 `ADR-NNN-kebab-case-title.md`（NNN 为三位数字编号，标题用小写连字符）。

3. **每个独立 ADR 必须遵循三段格式**：背景（为什么需要这个决策）、决策（具体决定了什么）、后果（决策带来的可观察影响 / 已做的验证 / 后续约束）。

4. **修改或废止 M0 基线 ADR 时**，新写一个独立 ADR（沿用规则 2~3），并在 ARCHITECTURE.md §3 对应原 ADR 末尾追加一行 `> 此决策已被 ADR-NNN <title> 更新/废止，见 docs/adr/`。原内联条目不删除，保留历史可追溯性。

## 后果

- ADR-001~005 不抽取——承认 M0 既定事实，避免一次性 5 文件 refactor 引入的合并风险与 M0 历史 commit 的可追溯性损失。
- ADR-006 / ADR-007 / ADR-008 的独立文件状态成为后续 ADR 的实践标准。
- 不立即创建 `docs/adr/README.md` 索引——`docs/adr/` 目录的 `ls` 输出即为天然索引，且 ARCHITECTURE.md §3 仍是基线 ADR 的入口。如未来 ADR 数量超过约 15 个、人工查找成本上升，可独立追加 ADR 决议是否引入索引文件。
- 本规则覆盖范围：仅 TerraWorks 单仓库内的 ADR 管理，不试图作为通用 ADR 最佳实践推广。
